Mapping the Performance-Privacy Frontier of Decomposed Zero-Knowledge Proofs
     Flat, Hierarchical, and Recursive Architectures for the Travelling Salesman Problem

## 1. What this project is

An MSc Cybersecurity thesis that **maps the performance/privacy trade-off space of
zero-knowledge proofs for the Travelling Salesman Problem (TSP)**.

The prover wants to convince a verifier that it knows a **Hamiltonian cycle** on a complete
weighted graph whose **total cost ≤ a public threshold T**, *without revealing the route*.
The thesis builds a family of ZK circuits that prove exactly this statement under different
**decomposition strategies**, benchmarks them, and extracts a small set of **structural
findings** that generalise beyond TSP.

The deliverable is **not** "I built hierarchical TSP-ZK variants" (that uses only standard
techniques: Merkle commitments, grand products, blinded commitments, in-circuit recursion).
The deliverable is **the map**: a framework explaining *why* decomposition behaves the way it
does in ZK, with the variants as the evidence that instantiates it.

### The four contributions (the spine of the whole thesis)

1. **The dualism** — Decomposing a *non-local* problem like TSP gives **no ZK speedup on
   total work**. Decomposition helps *classical* optimisation (it shrinks the search space),
   but ZK does no searching — it only *checks* a witness. The partition check that the
   recombiner must restore to stay sound *exactly cancels* the per-segment saving. The only
   real payoff is **parallelism** and **lower per-prover memory** (the same work, spread
   across machines). A negative result with a precise structural cause (the NP
   finder/checker asymmetry).

2. **The stitching tax** — Recombining K independent segment-proofs into one sound, private
   statement is a single artifact with **three coupled symptoms**: (a) **partition leakage**,
   (b) **O(K) verifier cost**, (c) **verifier-side bookkeeping**. They are not three problems
   but three faces of "independent proofs need external binding," and they dissolve *together*
   when the binding is folded into a proof (recursion). The tax is generated by **two
   decisions**: *where* the stitch is enforced (external / in-circuit / deferred) × *what* is
   stitched (plaintext / hiding commitment / witness).

3. **The frontier (pick-two triangle)** — Three goods: **cheap** (no more prover work than
   decomposition forces), **parallel** (independent segments across machines), **structurally
   private** (partition kept off the public surface, hidden as flat hides it). Each built
   architecture gives **exactly two**:
   - **flat** = cheap + private, *can't split*;
   - **committed** = cheap + parallel, only *computational* privacy;
   - **recursion** = parallel + private, *most expensive*;
   - **folding** (future) = the predicted empty corner that would break the triangle.

   Verifier cost is **not** a fourth axis: O(1)-vs-O(K) tracks the same external/internal
   `where` decision as privacy, so it rides with structural privacy.

4. **The methodology** — Controlled comparison: change **exactly one variable per step**;
   compare cost only at **equal privacy**; read cross-privacy deltas as "the price of hiding
   more" (the frontier), never as a ranking. The **2×2 factorial** {flat, recursive} ×
   {sort, product} de-confounds the *mechanism* axis from the *aggregation* axis and shows
   they are separable (≈ additive, no interaction) — the dualism, quantified.

### The second structural finding — the fingerprint lever

A subordinate result. Decomposition *forces* each segment to publish a **partition
fingerprint**. The naïve choice is the **sorted node set** (inherited from flat's sort check),
but it costs **O(M) public surface per segment** and a **serial O(N) merge** in the
recombiner. The **grand product** is the lever on that symptom: it collapses each segment's
shared surface from **O(M) → O(1)** (one field) and is **distributable**. **The price falls
out, not sought:** the exact, structural multiset check becomes a *probabilistic* one
(Schwartz–Zippel over a Fiat–Shamir challenge, sound only in the random-oracle model).
**Structural soundness → computational soundness is the cost of the surface win.** The same
lever is read four times down the stitching axis, and *means something different each time*
(see §4.4).

## 2. The problem, formally

- **Graph:** complete, directed, weighted on `n` nodes, given by an `n × n` integer cost
  matrix. (The undirected case is a special instance — symmetric matrix.)
- **Valid route:** a Hamiltonian cycle — an ordered sequence `[v₀, …, v_{n-1}]` of all `n`
  nodes, each once; the closing edge `v_{n-1} → v₀` is implicit.
- **Feasible route:** total cost `Σᵢ cost(vᵢ, v_{(i+1) mod n}) ≤ T`. **Feasibility, not
  optimality** — we prove "a route under budget exists", never "this is the minimum".
- **Relation `R`** proven in ZK: *"∃ a Hamiltonian cycle on the committed matrix with cost ≤
  T."*
- **Public inputs:** the cost matrix (flat-full) or a commitment to it (flat-Merkle), and `T`.
- **Private witness:** the cycle (and, per variant, Merkle openings / segment data).
- **The bottleneck that motivates everything:** prover-side **peak memory** is the wall. As
  `n` grows the single monolithic circuit becomes infeasible to prove — which is exactly what
  decomposition into K segments attacks (per-prover memory ≈ flat/K).

**Threat model:** a cheating, polynomial-bounded prover; an honest verifier. Guarantees:
completeness, knowledge soundness, zero-knowledge. **Non-guarantee:** input *validity* — that
the committed matrix is the *real* one — needs an external anchor (binding ≠ authenticity).
Side channels out of scope.

## 3. The variant grid — the in-scope construction set

Every variant is named **`{regime}-{mechanism}`**. The name pins its cell in a 2-axis grid.

| Stitching regime (↓ = the stitching tax) | **sort** mechanism | **product** mechanism |
|---|---|---|
| **flat**, monolithic (K=1) | `flat-sort` | `flat-product` *(control)* |
| **plain** stitch (hierarchical) | `plain-sort` | `plain-product` |
| **committed** stitch (hierarchical) | `committed-sort` | `committed-product` |
| **recursive**, in-circuit stitch | `recursive-sort` *(control)* | `recursive-product` *(shipped)* |

- **Vertical axis = the stitching tax** (*where/what is stitched*). The focus, the privacy
  axis, the frontier. Top → bottom: flat → plain → committed → in-circuit.
- **Horizontal axis = the fingerprint lever** (*how the partition is checked*): deterministic
  **sort** vs probabilistic **grand-product + Fiat–Shamir**. A *consequence* of the vertical
  axis, not co-equal.

**Comparison discipline (the methodology made literal):**
- **Down a column** = the stitching tax (mechanism fixed → a cost delta is pure aggregation).
- **Across a row** = the fingerprint lever (stitch fixed → pure sort-vs-product).
- **The diagonal is forbidden** (e.g. `flat-sort ↔ recursive-product` confounds *both*).
  Naming this as off-limits is half the methodology contribution.

**Out of scope** (kept in the repo, mentioned only as alternatives tried): `flat_full_*`
(pairwise/presence/invperm), `flat_merkle_presence`, and the disclosing-sub-matrix "Variant B"
(never built). `flat-product` and `recursive-sort` are **instrumental controls** for the Ch 6
factorial, off the frontier figure.

### Naming — three levels (reused verbatim everywhere)

- **Level 1 — variant:** `{regime}-{mechanism}`, regime ∈ flat·plain·committed·recursive,
  mechanism ∈ sort·product.
- **Level 2 — component:** `segment` (one M-node leg, the unit of parallelism) · `glue` (a
  *separate* recombiner proof checked alongside the K segments; used by plain-* / committed-*)
  · `outer` (a *recursive* proof that verifies the K segments **in-circuit**; used by
  recursive-*; the segments it consumes are its **inner** proofs). `recursive-*` reuses the
  `plain` segments as inners — no distinct recursive segment.
- **Level 3 — what each segment publishes:** node-set fingerprint `sorted_nodes[M]` (*-sort) ·
  product fingerprint `P_i` + chain anchors `h_in/h_out` (*-product) · segment commitment
  `C_i` (committed-*) · standing objects `start`/`end`/`partial_cost`.

### Old ↔ new ↔ code mapping (code identifiers are **unchanged** from earlier naming)

| New variant | Old name | Segment circuit dir | Recombiner circuit dir |
|---|---|---|---|
| `flat-sort` | flat_merkle_sort | — | `circuits/flat_merkle_sort` |
| `flat-product` | flat_merkle_grand_product | — | `circuits/flat_merkle_grand_product` |
| `plain-sort` | A | `hierarchical_segment` | `hierarchical_glue` |
| `plain-product` | A++ | `hierarchical_segment_fs` | `hierarchical_glue_fs` |
| `committed-sort` | committed-A | `hierarchical_segment_c` | `hierarchical_glue_c` |
| `committed-product` | committed-A++ | `hierarchical_segment_cfs` | `hierarchical_glue_cfs` |
| `recursive-sort` | recursive-A | *(reuses `hierarchical_segment`)* | `recursion` (sort inner) |
| `recursive-product` | recursive-A++ | *(reuses `hierarchical_segment_fs`)* | `recursion` (product inner) |

## 4. The constructions in detail

### 4.0 Shared flat anatomy — three constraint groups

Every circuit shares a thesis-facing **three-group skeleton** (the in-code comments still say
`GROUP 1..4/G1..G7`; the prose taxonomy is three groups):

1. **Permutation** — the cycle is a permutation of `{0,…,n-1}` (range check subsumed here).
2. **Edge cost** — each consecutive edge's cost is the real matrix entry (direct ROM lookup in
   flat-full, or a Poseidon2 Merkle opening against the committed root in flat-Merkle).
3. **Threshold** — the costs sum to `≤ T`.

The recurring design *pattern*: **the prover computes something off-circuit; the circuit checks
a cheap relation.** Examples: `≠`-via-modular-inverse; the inverse-permutation witness; the
sorted array verified by `check_shuffle`; the Merkle opening.

**Five permutation mechanisms** were built at flat (the §4.2 mechanism study). Only the first
and last are carried forward:

| Mechanism | Idea | Cost | Memory model |
|---|---|---|---|
| pairwise | `vᵢ ≠ vⱼ` ∀ i≠j via modular inverse | O(N²) | static |
| **sort** ✓ | sort cycle, assert `== [0..N-1]` (`check_shuffle`) | ~3N | **dynamic ROM** (~2N data-dependent reads) |
| invperm | explicit inverse-perm witness; N range + N ROM | 2N | ROM |
| presence | mutable `seen[]` boolean mark array | ~4N | **read-write RAM** |
| **grand-product** ✓ | in-circuit Fiat–Shamir `X=Poseidon2([c],1)`, then `∏(X+cycleᵢ)==∏(X+j)` | ~N Poseidon2 + 2N mults | **static** straight-line |

The **sort↔product pairing** is the thesis's sharpest contrast and reappears at every regime.

### 4.1 Matrix representation — flat-full vs flat-Merkle

- **flat-full:** all `n²` edge costs are **public inputs**. Simplest, no hashing; the matrix is
  fully disclosed.
- **flat-Merkle:** the matrix is committed to a single **Poseidon2 Merkle root** (1 public
  field); the prover supplies a per-edge **opening proof** (siblings + path bits) as private
  witness. **Soundness-critical leaf-index check:** path bits must reconstruct to exactly
  `from*N + to`, *and* the hash chain must reach the root — together they bind each cost to the
  committed entry. Explicit leaf-index check makes **leaf domain-separation redundant** (saves
  N Poseidon2 calls).
- There is an **N≈175 crossover** where flat-Merkle's per-edge hashing overtakes flat-full's
  raw `n²` public-input cost.
- **`flat-sort` is crowned the monolithic perfect-hiding baseline** the decomposition chapters
  answer to (public surface = `{root, T}`).

### 4.2 The hierarchical (glued) constructions

Decompose the N-node cycle into **K segments** of `M = N/K` nodes. Produce **K segment proofs +
1 glue proof**, all independent UltraHonk proofs. The verifier runs `bb verify` K+1 times *and*
a Python cross-check (`verify_hier*.py`) that the shared public-input fields actually agree
(same root, partition stitches, endpoints chain, partial costs sum ≤ T). Verifier-side cost is
O(N) trivial equality — negligible cryptographically, but it **expands the trusted-code
surface** and is **O(K)** in proof bytes / verify calls (the stitching tax, made operational).

- **`plain-sort` (the diagnosis):** each segment publishes its **sorted node set** + endpoints
  + partial cost in plaintext; the glue **sorts the concatenated N nodes** (the multiset
  check), verifies K boundary-edge Merkle proofs, sums ≤ T. **Deterministic.** Cheapest total
  gates of the family. **The partition is disclosed** — the tax, raw. The glue's O(N) sort is a
  serial memory floor (~159 MB at N=480).

- **`plain-product` (the lever, live):** sort/O(M) → grand-product/O(1). Each segment derives an
  FS challenge `X = Poseidon2([c],1)` (bound to the full-cycle hash-chain terminal `c`),
  publishes one product fingerprint `P_i = ∏ⱼ(X + node_j)` plus chain anchors `h_in/h_out`; the
  glue combines the K fingerprints (`∏ Pᵢ == ∏ⱼ(X+j)`, O(K) not O(N)). **Not cheaper** — it
  *relocates* the O(N) work into the segments. Justified on **surface + soundness + the
  recursion bridge, never on hiding** (it still leaks: `P_i` is a multiset oracle ~C(N,M),
  chain anchors an ordering oracle ~(M-2)!). **F7 (a key result): neither `plain-sort` nor
  `plain-product` dominates** — `plain-sort` is cheaper on *total* gates and deterministic;
  `plain-product` has the compact O(1) surface and is the design you can recurse on.

- **`committed-sort` / `committed-product` (the twins — the cure):** plaintext → **blinded
  commitment**. Each segment publishes one `C_i = fold(r_i, [values…])` (a Poseidon2 fold with
  blinding `r_i` from `/dev/urandom`); the glue takes values + openings as *witness* and
  re-checks in-circuit; the verifier cross-check collapses to **opaque-blob equality**
  (`sub_i.C_i == glue.C_i` — ZK for free). The leak closes: public surface is `{root, T, C_is}`
  (+ `X` for product). **Equal-privacy finding:** both twins reach the *same* privacy rung
  (multiset computational, interior order info-theoretic, reveal K commitments) — so here the
  sort↔product lever is a **privacy-neutral pure cost/soundness choice**. Pedersen is the
  analytical unconditional-hiding upgrade (left un-built).

### 4.3 The recursive construction (the climax)

`circuits/recursion` — the **outer** circuit. It takes the K `plain-product` segment proofs and
their public inputs **as private witness**, **verifies them in-circuit** (`verify_honk_proof`
over `UltraHonkZKProof`), then re-runs the glue logic on those now-trusted-but-private values.
The partition becomes a **witness** — it never reaches the outer proof's public surface, which
**collapses to `{root, T}`**: the **perfect-hiding endpoint, equal to flat**. The verifier runs
**ONE `bb verify`, zero cross-checks** — the stitching-tax collapse made operational.

- **Why `plain-product` is the inner** (the Chekhov's gun firing): the recursive verifier's
  cost is dominated by absorbing the inner's *public* inputs, so the inner must keep an O(1),
  **M-independent** public surface for the outer to stay flat at **~704k gates whether N=8 or
  N=480**. `plain-product`'s one-field `P_i` is exactly that. An `plain-sort` inner exposes O(M)
  and would make the outer grow with N. This is *why `recursive-product` is shipped*;
  `recursive-sort` is the off-frontier control.
- **Cost shape:** K in-circuit verifications (~704k gates each) dominate → total ≈ K × one
  verification + a small glue tax. This **buys back the O(K) verifier tax** (verifier checks one
  proof) at the price of ~704k-per-segment prover cost — recursion's *own* stitching tax (the
  Fiat–Shamir floor: in-circuit recomputation of the inner transcript).
- **ZK guard:** default `bb prove` is ZK (458-field / **14656-byte** proof vs 410 fields for
  `-no-zk`). The harness and verifier **assert the 14656-byte length** so a future bb default
  flip can't silently break hiding.

### 4.4 The lever's four readings (why it's not repetition)

| Regime | What the sort↔product lever means here |
|---|---|
| **flat** | Pure mechanism cost (witness-time inversion). Surface benefit = 0 → product is a *strictly worse* trade → the clean control. |
| **plain** | Lever goes live: O(M)→O(1) surface + distributable, paid in soundness. **F7: neither dominates.** |
| **committed** | **Equal-privacy:** both reach the same rung → lever is a privacy-neutral cost/soundness choice. |
| **recursive** | Hiding off the table (both partitions are witness) → lever is soundness-only + outer-surface: O(1) inner keeps the outer flat at ~704k. **Why product is shipped.** |

## 5. The software stack

| Layer | Choice | Notes |
|---|---|---|
| **ZK DSL** | **Noir** `nargo 1.0.0-beta.20` | high-level circuit language → ACIR bytecode |
| **Backend** | **Barretenberg `bb` 5.0.0-nightly.20260324, UltraHonk** | universal SRS (no per-circuit trusted setup); Plonkish + lookups; native recursion via `bb_proof_verification` |
| **Hash** | **Poseidon2** (`poseidon` Noir lib v0.3.0; `Poseidon2::hash([l,r],2)`) | ZK-friendly (~87 amortised gates/call measured via Plookup, vs ~264 literature); used for Merkle tree, FS challenge, hash-chain, and commitment fold |
| **Off-circuit hashing / tree build** | **Rust** crate `pipeline/merkle_builder/` | uses `bn254_blackbox_solver` + `acir` from the noir monorepo (tag `v1.0.0-beta.20`) so the `FieldElement` type matches Noir exactly; reads JSON (matrix+cycle) on stdin, writes each circuit's `Prover.toml` with all Merkle/chain/commitment witnesses |
| **Orchestration** | **Python** (conda env `zk-tsp`) | instance gen, solver, input formatting, benchmark harnesses, aggregation, plotting, cross-checks. Python **never hashes** — all Poseidon2 is in Rust or Noir |
| **Solver** | nearest-neighbour + 2-opt (pure numpy) | path quality irrelevant — we only need a *valid* cycle; OR-Tools dropped (removed `pywraprouting`) |
| **Thesis** | **Typst** (`thesis/template/`) | 7 chapters; `refs.yaml` bibliography; cross-ref ledger |
| **Version control** | git, remote `github.com:lcalllegari1/MSc-Thesis.git` | |

**Cross-language hash compatibility** is pinned by `tests/hash_compat/` (Rust ↔ Noir produce
identical Poseidon2 outputs for both 2-input compression and single-input hash) — load-bearing,
because the Rust builder and the Noir circuit must agree byte-for-byte on every hash.

### Why these choices (rejected alternatives)

- Noir+UltraHonk over **arkworks** (too low-level for the timeline) and **Halo2** (API
  complexity not justified before needing native recursion).
- Poseidon2 over SHA-256 (~264 vs ~28k gates/call).
- Hamiltonian **cycle** (standard TSP) over path.
- **Folding** (Nova/ProtoStar) is the explicit future-work corner — a *different backend*, out
  of this single-backend comparison by design; it would break the pick-two triangle.

### Circuit conventions

- Array sizes must be **comptime** in Noir, so each `N` is a **separate compiled circuit**.
  Harnesses patch `global N: u32 = …;` (and `M`, `K`, `DEPTH = ceil(log2(N²))` as applicable)
  with `sed`, then recompile.
- The committed N=8 defaults compile out-of-the-box for every circuit **except** `recursion`
  (its `Prover.toml` is harness-generated — it needs live inner proofs — and is git-ignored).
- `circuits/*/target/` and `data/instances/` are git-ignored (build artifacts / large trees).

## 7. Experimental method and data

### Metrics (per `(variant, n, run)`, CSV schema)

`variant, n, run, circuit_size, acir_opcodes, compile_s, witness_s, prove_s, verify_s,
proof_bytes, peak_mb`

`circuit_size` (UltraHonk gates) is the primary complexity metric. `acir_opcodes` is flagged in
the thesis as a **misleading** metric (the N≈30-vs-N≈175 discrepancy — "sanity-check what the
number means").

### Why one instance per N is sound

SNARK proving is **data-oblivious**: for a fixed circuit, gate count / proof size / peak memory
are independent of witness *values*; only timings vary, with system noise. So `instance_cache.py`
builds one `(instance, cycle, Poseidon2 tree)` per `(N, seed)` under `data/instances/`, shared
across **every variant, K, and run**. The Rust builder loads `tree.bin` via `--tree-cache`
(checksum-validated) instead of rebuilding the O(N²) tree. Instance *diversity* lives in
`tests/correctness/`, not the benchmark.

### Sweep frame (fixed once)

| Parameter | Value | Why |
|---|---|---|
| N-sweep | `48 96 192 480` (hier); flat to ~1000–5000 | all divisible by lcm(2,4,8)=8 |
| anchor N | **480** | fixed-N decomposition & frontier figures |
| K set | `2 4 8` | where the variants diverge |
| flat comparator | `flat_merkle_sort` at the **same** N | sort matches the hier permutation check |
| runs | 3 (flat/hier), 1–2 (recursion) | mean±std |
| deployment | concurrent **and** `--isolated` | two *different claims*, not two estimates |

### The three deployment readings (carry the labels into every figure)

| View | Claim |
|---|---|
| concurrent + parallel(max) | single-machine wall-clock of the decomposed job |
| **isolated + parallel(max)** | distributed wall-clock, one node/segment — **compute-only upper bound** |
| isolated + total(sum) | total CPU work — the dualism (work conserved, not reduced) |
| peak_mb (max) | per-node footprint (~1/K) — valid even from concurrent |
| verify_s / proof_bytes | O(K) stitching tax (hier) vs O(1) (recursion/flat) |

### Headline measured results

- **K× speedup is MEASURED** (not projected): at **N=3000, K=8**, critical path =
  max-segment + glue. `plain-product`: 13.2 s vs flat 86.1 s → **~6.5×**. `plain-sort`:
  19.7 s vs 91.9 s → **~4.7×** (its serial O(N) glue sort is the gap to ideal).
- **Witness-time inversion** (the flat-row finding): `flat-product` compiles to **more**
  gates (+3.5–7.9%) yet solves its witness **faster** (up to **−44%** at N=1000) — the sort's
  `check_shuffle` does ~2N dynamic-ROM reads; the product is straight-line static arithmetic.
  Both land in the same dyadic bucket → identical proof size (14656 B), prove time, memory.
  *"Gate count and witness cost are different currencies."*
- **Recursion outer ≈ 704k gates, N-independent** (704,363 at both N=8 and N=480), ~K× total.
- **flat-full ↔ flat-Merkle crossover at N≈175**; ~87 amortised gates/Poseidon2 call.

## 8. Correctness / soundness validation

Each circuit has a `tests/correctness/test_*.py` suite that asserts the circuit **rejects every
invalid witness** (`nargo execute` exits non-zero when an `assert` fires). Categories: valid
baseline · out-of-range node · duplicate nodes · threshold below cost · tampered cost matrix ·
edge cases. Merkle suites add: tampered `edge_costs`, flipped path bit, wrong root. Hierarchical
suites add the partition-overlap (Schwartz–Zippel) negative and the verifier cross-check
rejecting mixed proof sets. Committed suites add sub-side (G8) and glue-side (G0) commitment
binding. Recursion suite (9/9): tampered-proof and tampered-VK at prove time, plus the in-circuit
asserts.

This is **validation, not proof.** The formal **knowledge-soundness reductions** (ε ≤ ε_SNARK +
ε_FS + ε_SZ + ε_bind, composing cited UltraHonk-KS / FS-in-ROM / recursion-KS rather than
reproving them) are deferred analytical work (thesis §5.11 / §6.7).

### Honesty refrains (repeat throughout)

- **Total work is conserved** — hierarchical never beats flat on total gates; the win is
  parallelisability.
- **`plain-product` is never motivated on hiding** — it still leaks (the `P_i` oracle).
- **"Perfect hiding" of flat/recursion is *structural*, not information-theoretic** —
  UltraHonk-ZK is computational/statistical and identical across all variants; it is *not* a
  discriminator. The public surface carrying no partition info is the whole claim.
- **The K× speedup is measured, with a stated deployment assumption** — per-proof times are
  uncontended single-machine; K-machine wall-clock = max(segment) + glue.

## 10. How to recreate the project from scratch — ordered checklist

1. **Environment.** Install `nargo 1.0.0-beta.20`, `bb 5.0.0-nightly.20260324`, Rust, and a
   conda env `zk-tsp` (Python + numpy + matplotlib). `conda activate zk-tsp`.
2. **Build the Rust builder.** `cargo build --release --manifest-path
   pipeline/merkle_builder/Cargo.toml`.
3. **Pin the hash.** Implement/port `tests/hash_compat/` first — every later circuit depends on
   Rust and Noir agreeing on Poseidon2. Don't proceed until it passes.
4. **Flat-full baseline.** `instance_gen.py` → `solver.py` → `format_inputs.py` →
   `flat_full_pairwise`. Add the other four mechanisms (sort/invperm/presence/grand-product) —
   this *is* the §4.2 study and the witness-inversion data.
5. **Flat-Merkle.** Add the Rust tree build + the per-edge opening with the **leaf-index
   check**. `flat_merkle_presence`, then `flat_merkle_sort` (= `flat-sort`, the crowned
   baseline) and `flat_merkle_grand_product` (= `flat-product`).
6. **Plain hierarchical.** `hierarchical_segment` + `_glue` (`plain-sort`), then `_fs`
   (`plain-product`, grand-product + in-circuit FS). Add `run_hier*.py`, `verify_hier*.py`,
   `aggregate_hier.py`, and the correctness suites.
7. **Committed hierarchical.** `_c` / `_cfs`: add the blinded Poseidon2 commitment fold, the
   in-circuit glue re-check, and the opaque-blob cross-check.
8. **Recursion.** `circuits/recursion` over the `plain-product` segments; `run_recursion.py`
   assembles the outer `Prover.toml` from live inner proofs; `verify_recursion.py` asserts the
   14656-byte ZK proof. Verify the outer is ~704k gates and N-independent.
9. **Instance cache.** Add `instance_cache.py` + `merkle_builder --tree-cache` so sweeps reuse
   one tree per N.
10. **Sweeps.** Run the §7 funnel: Phase 0 gates (deterministic) → Phase 1 concurrent → Phase 2
    `--isolated` (the K× claim) → Phase 3 recursion (RAM-gated: K=2≈2GB, K=4≈4GB, K=8≈8GB+) →
    Phase 4 aggregate. Gate every cell on `xchecks_ok==1` + all `bb verify` passing.
11. **Figures + thesis.** `plot_thesis_figures.py` → the 12 figures; write the 7 Typst chapters
    against `NAMING.md` + `STYLE_GUIDE.md`.
