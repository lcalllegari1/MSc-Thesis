This chapter introduces the problem, beginning with a real-world scenario. We then
present a graph-based framework for its formal definition and conclude by developing a
mathematical model that specifies the objective and the constraints.


1.1 Problem Description
A company manufactures perforated boards used in the assembly of electrical panels. During
production, each board is mounted on a drilling machine where a drill head moves across
the surface, stops at predefined positions, and drills the required holes. Once a board is
completed, it is replaced by the next one, and the process repeats continuously.
   Given a set of hole positions on a board, the objective is to determine an ordering
of drilling operations that minimizes the total processing time. Since the drilling time at
each hole is assumed to be constant, minimizing the overall processing time reduces to
minimizing the travel time of the drill head between successive hole locations.
   The travel time between two holes depends on the motion characteristics of the drilling
head. In particular, the machine may operate at different maximum speeds along each axis,
and the way movements along the axes are coordinated directly affects how long it takes
to move from one position to another. As a result, travel times cannot be inferred solely
from geometric distances, but must instead be computed explicitly for each ordered pair
of holes.
   The kinematic configuration of the drilling machine therefore induces a specific distance
metric used to evaluate travel times. In this work, we consider three representative
movement modes:

   1. Sequential axis movement (𝐿1 norm): the drill head moves along one axis at a time.
      In this case, the travel time is proportional to the Manhattan distance between two
      points. This mode reflects a system in which motion along one axis precludes motion
      along the other.
   2. Coordinated straight-line movement (𝐿2 norm): the drill head follows a straight-line
      path between source and destination. This corresponds to the Euclidean distance,
      under the assumption that maintaining a fixed movement direction requires sharing
      power between axes, thereby limiting achievable speed.
   3. Simultaneous independent axis movement (𝐿∞ norm): the machine is equipped with
      independent motors that can operate simultaneously at their maximum speeds along
      both axes. In this case, the travel time is determined by the axis requiring the longest
      time to complete its motion, corresponding to the Chebyshev distance.

1.2 Mathematical Formulation

1.2.1 Modelling Framework
The generic instance of the problem can be modeled as a weighted complete graph, denoted
as 𝐺 = (𝒩︀, 𝒜︀), where 𝒩︀ is the set of nodes representing the holes that the drilling machine
must process, and 𝒜︀ is the set of arcs connecting each pair of nodes.
   Formally, we can define 𝒩︀ = {0, 1, …, 𝑛 − 1} and 𝒜︀ = {(𝑖, 𝑗) : 𝑖, 𝑗 ∈ 𝒩︀ ∧ 𝑖 ≠ 𝑗}, with a
function 𝑐 : 𝒜︀ → ℝ that maps (𝑖, 𝑗) ∈ 𝒜︀ ↦ 𝑐(𝑖, 𝑗), representing the cost of the trajectory
between nodes 𝑖 and 𝑗. This cost can be interpreted as the time required for the drill head
to move from 𝑖 to 𝑗, i.e. 𝑐(𝑖, 𝑗) = time(𝑖 → 𝑗). The condition 𝑖 ≠ 𝑗 for the set 𝒜︀ of arcs
naturally arises in the context of our problem, since we can exclude loops, i.e. arcs that
go from one node to itself. This is reasonable because the drill head moves between nodes,
with no benefit gained from remaining at the same position.

1.2.2 Problem Definition
With the framework established, we can now proceed to formally define the problem.

 The Travelling Drillman’s Holey Grail
    Let 𝐺 = (𝒩︀, 𝒜︀) denote the graph representing an instance of our problem, in the
 form provided by the framework described in 1.2.1. Thus, solving the problem reduces
 to finding a minimum-cost path that starts at a node, visits all other nodes exactly once,
 and returns to the starting node. In other words, we seek a Hamiltonian cycle in 𝐺, which
 is equivalent to solving the Travelling Salesman Problem (TSP) on 𝐺.

In line with the model we have defined, we assume that the drill head returns to the initial
hole after completing all others. This assumption is reasonable, as it ensures that the drill
head is ready to start drilling the next board.

1.2.3 Model Definition
Several formulation for the TSP have been formulated over time using Mixed Integer
Linear Programming (MILP). Some of them include an exponential number of constraints
and in practice they cannot be directly implemented using basic CPLEX APIs. For this
reason, we consider a more compact formulation that requires a polynomial (possibly with
small degree) number of variables and constraints. More specifically, we consider a compact
formulation based on network flows (Single-Commodity Flow formulation).
   Given the graph 𝐺 = (𝒩︀, 𝒜︀) for an instance of our problem, we arbitrarily choose a
node in 𝒩︀ as the starting node (denoted as node 0 for simplicity). Now, let 𝑛 = |𝒩︀|, and

  2                                                 Instance Generation
This chapter presents the instance generation process, starting from the real-world problem
and progressively translating it into a formal representation. We first provide a high-level
description of the structure and parameters defining an instance, then detail how instances
are generated based on these parameters. Finally, we introduce the file format used to store
instances, ensuring consistency and ease of integration with the solver.


2.1 Creating Instances
To rigorously evaluate the proposed optimization model, we developed a custom instance
generator designed to reflect the structural characteristics of real industrial circuit boards.
Since there are infinitely many ways to generate drilling layouts, the generator enforces a
set of controlled constraints that balance realism, diversity, and reproducibility.
   Rather than modeling boards as unstructured collections of random points, instances
are constructed to mimic the spatial organization induced by electronic components. In
real boards, holes are rarely placed arbitrarily; instead, they appear in structured patterns
dictated by component footprints, mechanical constraints, and assembly requirements. The
generator explicitly incorporates this structure.

2.1.1 Configuration-Driven Design
The generation process is fully configuration-driven. Each dataset is specified through a
declarative configuration file that defines:

  • the physical dimensions of the board,
  • the safety constraints such as edge margins and minimum hole spacing,
  • the presence of fixed mechanical features (e.g. mounting holes),
  • and the types and quantities of components to be placed.

This approach enables reproducibility while allowing the systematic exploration of diverse
scenarios, ranging from sparse and loosely structured layouts to dense and highly organized
boards. As a result, the optimization algorithms can be tested across a wide spectrum of
instance characteristics.

2.1.2 Component-Based Construction
At the core of the generator lies a component-based constructive approach. Instead of
treating each hole as an independent entity, holes are generated in structured groups
corresponding to common component footprints. Each component defines a local geometric
pattern, which is then placed onto the board as a single unit. More specifically, the following
component primitives are supported:



                                               9
  • Grids: representing regular pin layouts such as those found in integrated circuits or
     sockets.
  • Linear arrays (straight lines): modeling connector headers or strip terminals.
  • Random holes: used to introduce controlled irregularity and represent discrete com­
     ponents such as resistors or capacitors.
  • and the types and quantities of components to be placed.
By assembling instances from these primitives, the resulting layouts preserve local spatial
correlations commonly observed in real production boards. This provides a significantly
more realistic challenge for drill-path optimization than purely random point sets.

2.1.3 Board Constraints
Before placing variable components, the generator defines the static constraints of the
physical board. A rectangular canvas with fixed width and height is established, together
with a safety margin that prevents drilling too close to the board edges, where mechanical
clamping or material instability may occur.
   Optionally, fixed mounting holes can be generated at predetermined locations such as
corners, edges, or the board center. These holes represent mechanical fixation requirements
and are treated as deterministic features, independent of the component placement logic.

2.1.4 Stochastic Placement and Feasibility Checks
Components are placed using a stochastic try-and-check procedure. For each component
instance, the generator:
  1. Constructs the component’s local hole pattern.
  2. Applies a rotation, either fixed or randomly sampled.
  3. Samples a translation within the valid board area.
  4. Verifies physical feasibility before final placement.
Feasibility is enforced through collision detection: the Euclidean distance between every
candidate hole and all previously placed holes must exceed a configurable minimum spacing
threshold. This threshold reflects the drill bit diameter plus a safety tolerance. If a placement
violates this constraint, it is rejected and resampled, up to as many times as specified by a
proper parameter, as to avoid infinite looping.
   This process ensures that all generated instances are not only mathematically valid, but
also physically manufacturable, avoiding unrealistically dense or overlapping layouts.


2.2 Storing Instances
After generating the board layouts, instances must be stored in a consistent and structured
format to allow reliable reuse. This is essential both for flexibility of the optimization
framework and for reproducibility, as the same instance can be processed multiple times to




                                                10
compute averages and obtain statistically robust results. The storage pipeline is organized
into two distinct stages.

2.2.1 Board Representation
In the first stage, each generated board is stored as a standalone JSON file. This file encodes
the geometric and structural description of the instance and includes:
  • the board dimensions and safety constraints,
  • the complete list of hole coordinates (𝑥, 𝑦) for the components that identify all holes
        on the board, and
  • metadata such as measurement units and unique instance identifiers.
At this stage, instances are purely geometric. The placement of holes is represented only by
their coordinates within the board, without any assumptions about the drill head’s motion
or travel-time model.

2.2.2 Cost Matrix Representation
In the second stage, a dedicated preprocessing step converts the geometric instance into a
weighted routing problem. Given a selected distance metric and the motion parameters of
the drill head, the processor computes the full travel-time cost matrix between all ordered
pairs of holes.
    The conversion from geometry to routing costs is performed with respect to the node
identifiers defined in the geometric instance. In the JSON representation, each hole is asso­
ciated with a unique integer id, which establishes a fixed ordering of nodes. This ordering
is preserved during preprocessing and used consistently to index the cost matrix: the entry
in row 𝑖 and column 𝑗 corresponds to the computed travel time from the hole with ID i to
the hole with ID j. In particular, node IDs are assumed to range from 0 to 𝑛 − 1, where 𝑛
is the number of holes, ensuring a one-to-one correspondence between geometric nodes and
matrix indices.
    This cost matrix captures all kinematic effects, including axis-dependent speeds, asym­
metries, and the chosen movement model. The resulting matrix is stored in a separate .dat
file.


2.3 Loading Instances
The optimization algorithm operates exclusively on the precomputed cost matrix and has
no access to geometric coordinates, board dimensions, or kinematic parameters. From the
solver’s perspective, the problem is therefore fully specified by the matrix alone, which
cleanly decouples instance generation, physical and kinematic modeling, and optimization.
This design allows different motion models or distance metrics to be evaluated on the same
geometric instance without modifying the optimization code, while also enabling the same
cost matrix to be reused across multiple experimental runs.

and optimization pipeline.
   Overall, this approach provides a high degree of flexibility: instances can be randomly
generated, systematically modified, or explicitly handcrafted, all while preserving repro­

  3                                          Model Implementation
This chapter describes the implementation of the Single-Commodity Flow (SCF) formu­
lation introduced previously, using the CPLEX C APIs. The goal is to translate the
mathematical model into an executable optimization program capable of solving the
generated drilling instances.
   The focus is on modeling choices, variable construction and constraint encoding. Low-
level details of the CPLEX API are omitted where they do not contribute to understanding
the model implementation, but they are still available for inspection in the codebase.


3.1 Input Interface and Data Preparation
The solver operates on a two-stage input pipeline, as explained previously. In the first stage,
the instance generator produces the board layout and stores the hole coordinates in a JSON
file. In the second stage, a dedicated processor computes the travel-time cost matrix based
on the selected distance metric and motion parameters, producing a standalone .dat file.
The solver reads this .dat file as input, which begins with a single integer specifying the
number of holes, followed by rows of floating-point values representing the travel-time costs
between each ordered pair of holes. This design ensures that the optimization algorithm
interacts only with the numeric cost matrix, with no interest about distance metrics or
motion parameters.
   The format of the traditional .dat file is shown in the example below:
                                         instance.dat
                                         5
                                         0 1 9 9 9
                                         9 0 1 9 9
                                         9 9 0 1 9
                                         9 9 9 0 1
                                         1 9 9 9 0

This is naturally a trivial example, but it effectively illustrates the structure of the .dat
file. From the way the cost matrix is populated, the optimal solution can be immediately
identified by inspection. Assuming holes are numbered from 0 (the starting hole) and follow
the order in the cost matrix, the resulting trivial sequence of drilling operations is:
                                  0 → 1 → 2 → 3 → 4 → 0.

When the .dat file is read, the travel-time values are stored internally in a one-dimensional
array rather than a full two-dimensional matrix. This choice improves memory efficiency
and access speed for large instances. To maintain clarity and usability, the implementation
provides access routines that emulate standard matrix indexing. After this phase, the solver
will only need to use the stored values of the cost matrix to proceed in the solve.

The Single-Commodity Flow (SCF) model presented previously is implemented using the
CPLEX C APIs, which require encoding the problem in a specific linear form. Constraints
are expressed as linear equations or inequalities, and the variables must be properly aligned
to their coefficients for the model to be correctly encoded. Additionally, for efficiency reasons
(especially when handling large instances), CPLEX requires that the specification of the
constraints is provided in a compressed sparse row (CSR) format.
   With this framework in place, we now proceed to the implementation of the SCF model,
beginning with the definition of decision variables and the objective function. The code
snippets provided below are partial and do not include the declaration of all variables or

model into executable CPLEX calls, providing a clear mapping between the formal formu­
lation and its computational implementation. All auxiliary details are omitted for brevity,
clarity, and readability.

    The SCF formulation requires four families of constraints to properly model the routing
problem. These constraints enforce both the connectivity of the solution and the correct
interaction between the routing and flow variables:
  1. Flow conservation constraints, which ensure that the flow originating from the root
       node is correctly propagated through the tour.
  2. In-degree constraints, which guarantee that each node is entered exactly once (i.e.
       from a single arc of its incoming arcs).
  3. Out-degree constraints, which guarantee that each node is exited exactly once (i.e.
       through a single arc of its outgoing arcs).
  4. Linking constraints, which tie the routing variables 𝑦𝑖𝑗 to the flow variables 𝑥𝑖𝑗 ,
       ensuring consistency between the decision to traverse and arc and the flow passing
       through it.

abstract mathematical definitions are mapped into CPLEX API calls.

A key difference from the previous constraints is that each linking constraint involves
exactly two variables: one flow variable and its corresponding routing variable. There is no
summation over indices here, no accumulation across rows or columns. Each arc produces
one constraint, and each constraint touches precisely two columns of the coefficient matrix.
The right-hand side is set to zero and the constraint sense is 'L', corresponding to a “less-
than-or-equal” inequality. One last remark clarifies the role of the nested loops: they repeat
this constraint for every admissible arc (𝑖, 𝑗) ∈ 𝒜︀. Each iteration of the loops corresponds
to one distinct linking constraint and therefore to one distinct row added to the model.
   To summarize it all, the degree constraints shape the tour, the flow conservation
constraints enforce connectivity, and the linking constraints glue the two worlds together.
With all of them in place, the SCF formulation is complete, and the implementation now
matches the mathematical model.


3.3 Solution Extraction and Interpretation
Having defined and solved the model, we now address the final step of the pipeline:
extracting and interpreting the solution produced by CPLEX. Although the solver returns
values for all decision variables, only the routing variables 𝑦𝑖𝑗 are needed to reconstruct the
drilling order.
   CPLEX provides the solution as a flat array of variable values, indexed according to
the order in which variables were added to the model. The implementation must therefore
identify the entries corresponding to the routing variables, map them back to their (𝑖, 𝑗)
indices, and assemble a valid tour. The following snippet outlines the procedure to do so.


    1   size_t current = 0; std::vector<int> sequence; sequence.push_back(0);
    2
    3   while (true) {
    4       bool found = false;
    5
    6       for (size_t j = 0; j < n && !found; ++j) {
    7           const idx y_idx = m_ymap[current * n + j];
    8           if (y_idx == -1) continue;                   /* only consider existing vars */
    9
  10            if (all_vars[y_idx] > 1e-6) {
  11                current = j;
  12                sequence.push_back(j);
  13                found = true;
  14            }
  15        }
  16
  17        if (current == 0)
  18            break;
  19    }
  20
  21    return sequence;

                                                 25
The reconstruction procedure exploits a simple property of the SCF formulation: in a
feasible solution, each node has exactly one outgoing routing arc selected. As a consequence,
the active 𝑦𝑖𝑗 variables define a single directed cycle, which can be recovered by starting
from the source node and repeatedly following the selected outgoing arc until the tour closes.
   The code begins by initializing the reconstruction at node 0, which plays the role of the
reference starting point in the model. The vector sequence is used to store the ordered list
of visited nodes, and it is seeded with the starting node. At each iteration of the outer loop,
the algorithm scans all potential successors of the current node by using the lookup map to
translate a logical index pair into the corresponding index position in the linear array that
stores all the variables. When a routing variable with value greater than a small numerical
tolerance is found, the corresponding arc is interpreted as selected in the solution. The
algorithm then advances to the node just found, appends it to the sequence, and continues
the process from there in a new iteration. Since the model enforces exactly one outgoing
arc per node, the first such variable encountered is guaranteed to be the correct successor.
The outer loop terminates once the traversal returns to node 0, indicating that the cycle
has been closed and the full tour has been reconstructed.
   Having the solution as an explicit sequence of hole indices is extremely practical: it
provides a direct, human-readable representation of the tour, rather than a raw collection of
variable values. This format makes it trivial to visualize the path on the board, highlighting
the actual drilling order and enabling immediate inspection of patterns or inefficiencies.
Moreover, it facilitates quantitative comparisons: heuristic solutions can be directly aligned
with the optimal sequence, allowing for easy computation of deviations, travel-time differ­
ences, or other performance metrics.
   By converting the solver’s output into this structured form, we bridge the gap between
the abstract mathematical solution and the real-world object associated with an instance.


3.4 Running the SCF Solver
Now that the model has been fully implemented, we can turn to the actual execution of the
solver. At this stage, the C++ code acts as a self-contained program that reads a preprocessed
instance (the cost-matrix .dat file), builds the SCF model in CPLEX, and invokes the
optimization routine to find an optimal routing.
   For compilation and execution, we rely on a Makefile-based workflow. This ensures that
all source files, including the interface with the CPLEX C APIs, are compiled with the
correct flags, libraries, and include paths.

3.4.1 Compilation
Before running the solver, the system must be properly compiled. The code relies on a
C++ compiler supporting at least C++11, such as g++, and requires access to the CPLEX C

APIs headers and libraries. Compilation is handled through the provided Makefile. When
executed, the Makefile organizes the build process by first creating a build/ directory where


                                               26
all intermediate object files are stored, and then producing the final executables in the
bin/ directory. This structure keeps source and binary files separate and allows for a clean

rebuild.
   To compile the code, you can simply run the following commands.

  $ make clean     # safely removes old object files and binaries for a fresh rebuild
  $ make all


If everything compiles successfully, the bin/ folder will contain two executables. The first,
single_instance_cplex_solver, is designed to solve a single .dat instance at a time, allowing for

detailed inspection of the solution and intermediate outputs. The second, batch_cplex_solver,
is intended to process multiple instances sequentially, producing results suitable for aggre­
gate analysis and performance benchmarking across a dataset.
   With the compilation complete and the executables in place, we can now proceed to
demonstrate their use.

3.4.2 Solving an Instance
To solve a single instance, the single_instance_cplex_solver executable is employed. The full
usage is the following.

  $ ./bin/single_instance_cplex_solver <filename> [--verbose <0|1|2>] [--average <n_runs>]


The program expects the path to an input .dat file as a mandatory argument, which contains
the number of nodes and the cost matrix for the instance. Two optional flags can also be
provided: the first is --verbose, which controls the level of console output and defaults to
1 if omitted, and the second is --average, which specifies the number of independent solver

runs over which the results are averaged, with a default value of 5. Averaging multiple runs
is particularly useful for obtaining robust performance metrics and mitigating variability
due to solver heuristics or numerical factors.
   To take a concrete example, suppose we have an instance example.dat for which we want
the solution and the average solve time that results from the execution. Then we could run
the command that follows.

  $ ./bin/single_instance_cplex_solver example.dat --verbose 1 --average 10


Here we have required that the results come from averaging 10 runs of the solver on the
same instance. We can see that the solver takes the .dat file as input, which confirms that
it operates solely on the precomputed cost matrix. It does not have access to the original
board geometry, distance metrics, or kinematic parameters. This separation ensures that the
solver remains fully agnostic to how the instance was created, allowing the same executable
to handle any valid cost matrix without modification.
   If needed, the output of the solver can be redirected to a file using standard shell
redirection. This is particularly useful when running experiments or collecting results for



                                               27
post-processing and plotting. For instance, appending > results.dat to the command will
store all console output in a file instead of printing it to standard output.

3.4.3 Batch Solver
The batch solver provides an automated way to solve multiple instances in a single
execution. Instead of operating on a single .dat file, it takes as input a directory containing
a collection of instances and processes them sequentially using the same SCF model and
solver configuration. The command syntax is:

  $ ./bin/batch_cplex_solver <instances_folder> [--average <n_runs>] [--output <filename>]


The only mandatory argument is instances_folder, which must point to a directory contain­
ing the preprocessed .dat files to be solved. The optional --average parameter specifies the
number of repeated runs performed for each instance, allowing solution times and objective
values to be averaged for more robust performance evaluation. If omitted, a default value of 5
is used, consistent with the single-instance solver. The optional --output filename parameter
controls where the aggregated results are written; if this option is not provided, all results
are saved by default to a file named results.dat in the working directory.
   As a concrete example, assuming the instances are stored in a dedicated directory named
data/instances/, the batch solver can be invoked as follows:


  $ ./bin/batch_cplex_solver data/instances/ --average 10 --output batch_results.dat


This command processes all .dat files found in the data/instances/ directory, runs the solver
ten times per instance to compute averaged results, and writes the aggregated output
to batch_results.dat.
   Finally, all commands and paths shown above (for both solvers) assume execution
from the project’s root directory (i.e., where the Makefile is located) and must be adapted
accordingly if run from a different location.

  4                                                 Heuristic Algorithm
This chapter introduces a heuristic approach to the drilling optimization problem, focusing
on a two-opt tabu search metaheuristic. After presenting its core principles and implemen­
tation, we discuss how the specific structure of the problem can be exploited to improve both
efficiency and solution quality. These observations motivate the design and implementation
of enhanced variants of the basic method, leading to a more effective metaheuristic tailored
to the characteristics of the drilling path optimization problem.
   A brief disclaimer is in order. The final algorithm presented in this chapter is the result of
many hours of development, including numerous failed attempts and iterative refinements.
Over time, the method evolved into a sufficiently complex system that a fully detailed
description of every design choice and optimization is no longer feasible within the scope
of this report. Many small implementation details contribute to its overall efficiency, but
addressing all of them would require disproportionate space. For completeness and trans­
parency, the full codebase remains available for inspection and documents all algorithmic
details and implementation choices.


4.1 Local Search Algorithm Design
For the heuristic approach, we rely on a local search strategy based on the well-known
two-opt move, a standard technique in the context of the Traveling Salesman Problem.
Two-opt iteratively improves a candidate tour by removing two edges and reconnecting
the resulting paths in the alternative way. Despite its conceptual simplicity, this move is
particularly effective at eliminating crossings and reducing overall tour length, making it a
natural baseline for problems that involve finding efficient orderings over a set of locations,
such as the drilling sequence considered here.

4.1.1 Solution Representation
A solution, that is, a candidate drilling tour, is represented as an ordered sequence of
holes specifying the order in which drilling operations are performed. This representation is
particularly well suited to two-opt local search, since a two-opt move reduces to reversing
a contiguous subsequence of the tour, an operation that can be implemented efficiently.
   In the implementation, a solution is encapsulated in a Tour object, which is essentially a
lightweight wrapper around a std::vector storing the sequence of hole indices. The repre­
sentation is open, meaning that the tour is stored as a simple sequence visiting all nodes
exactly once, without explicitly repeating the starting node at the end. This design choice
simplifies the handling of two-opt moves, especially those involving the boundaries of the
sequence. Importantly, this does not alter the semantics of the solution: when evaluating
a tour, the closing edge from the last hole in the sequence back to the first one is always
taken into account. In other words, although the tour is represented internally as an open

                                              30
path, it is conceptually interpreted as a closed cycle, defined by the sequence together with
the implicit final edge connecting its endpoints.

4.1.2 Finding Initial Solutions
The initial solution for the local search is constructed using a nearest neighbor heuristic.
This choice is motivated by three main considerations. First, even a simple greedy
construction based on nearest neighbors produces solutions that are vastly superior to a
randomly generated tour, providing a much better starting point for local improvement.
Second, nearest neighbor is straightforward to implement: it builds a tour incrementally
by repeatedly selecting the closest unvisited hole, making the construction logic simple,
efficient, and easy to integrate into the overall framework. Third, and most importantly
in the context of a multi-start strategy, nearest neighbor naturally introduces diversity.
Because the greedy selection heavily depends on the starting node, initializing the algorithm
from different starting holes leads to distinct tours that tend to share locally good edges
but differ globally, which is desirable for exploration.
   Alternative construction heuristics, such as farthest insertion, were also considered due
to their better worst-case approximation properties and greater stability. However, these
methods tend to produce more similar initial tours across different runs, reducing diversity.
Given the intention to rely on multiple independent starts, we favored a simpler construction
heuristic that promotes variability among initial solutions, with the expectation that this
increased diversity would compensate for the slightly weaker average quality of the initial
tours once local search is applied.
   The code in Listing 9 illustrates the core logic of the nearest neighbor construction
heuristic used to generate an initial tour. The snippet shows how a feasible solution is built
incrementally, starting from a chosen hole and extending the tour one node at a time.


  1   /* Add the starting node to the tour */
  2   size_t curr_idx = start_idx;
  3   tour.push_back(nodes[curr_idx]);
  4   visited[curr_idx] = 1;
  5

  7       curr_idx = find_nearest_unvisited(nodes[curr_idx], nodes, visited, costs);

The procedure begins by inserting the selected starting node into the tour and marking it
as visited. At each subsequent step, the algorithm identifies, among all unvisited nodes, the
one that is closest to the current position according to the precomputed cost matrix. This
node is then appended to the tour and marked as visited. The process repeats until all holes

                                                31
have been included exactly once, resulting in a complete open sequence that represents a
valid tour.

4.1.3 Defining the Neighbourhood of a Solution
In the context of our heuristic, the neighborhood of a solution is defined by all tours that
can be obtained from the current one through a single two-opt move.
   Given a tour represented as an ordered sequence of 𝑛 holes, a two-opt move selects
two distinct indices 𝑖 < 𝑗 and reverses the subsequence between them. In terms of the
solution representation, this operation removes the two edges (𝑣𝑖−1 , 𝑣𝑖 ) and (𝑣𝑗 , 𝑣𝑗+1 ) and
replaces them with (𝑣𝑖−1 , 𝑣𝑗 ) and (𝑣𝑖 , 𝑣𝑗+1 ), respectively, implicitly reconnecting the tour
while preserving feasibility. Because the tour is stored as an open sequence, the reversal is
implemented directly on the underlying vector, while the closing edge between the last and
first elements is still accounted for during evaluation. Enumerating the full neighborhood
therefore amounts to iterating over all admissible index pairs (𝑖, 𝑗) with 0 ≤ 𝑖 < 𝑗 < 𝑛,
excluding degenerate cases that would leave the tour unchanged (i.e., 𝑖 = 0 ∧ 𝑗 = 𝑛 − 1,
which would just reverse the direction of the current tour making no difference in the cost
as long as symmetry is assumed).
   The size of this neighborhood grows quadratically with the tour length, on the order
of 𝒪︀(𝑛2 ), which is manageable for moderate instance sizes but large enough to motivate
efficient evaluation strategies and pruning when embedded in an iterative local search such
as tabu search.

4.1.4 Evaluating the Quality of a Solution
Evaluating the quality of a candidate solution amounts to computing the total travel time
induced by the corresponding tour. Given a tour represented as an ordered sequence of
holes, this is done by summing the travel times associated with each consecutive pair in
the sequence, plus the closing edge that connects the last hole back to the first one. Under
this representation, a full evaluation requires a single pass over the sequence and therefore
runs in linear time 𝒪︀(𝑛), where 𝑛 is the umber of holes in the (open) tour.
   However, for the purpose of local search, repeatedly recomputing the full cost would be
unnecessarily expensive. By assuming symmetric instances, we can exploit an important
structural property of two-opt moves: any two-opt transformation removes exactly two
edges from the current tour and replaces them with two new ones. As a consequence, the
cost of a neighboring solution can be obtained in constant time by adjusting the current
tour cost, subtracting the costs of the removed edges and adding the costs of the newly
introduced ones. This reduces the evaluation of a neighbor from 𝒪︀(𝑛) to 𝒪︀(1), which is
crucial when exploring a neighborhood of quadratic size.
   In practice, the algorithm does not even need to compute the absolute cost of a neighbor.
Instead, it is sufficient to compute the cost variation, or delta, associated with a given
two-opt move. This delta is defined as the difference between the cost of the candidate
tour and the cost of the current tour. Negative deltas correspond to improving moves,


                                              32
with more negative values indicating better improvements. By ranking candidate moves
according to their deltas, the heuristic can efficiently identify promising neighbors without
ever explicitly reconstructing or fully evaluating the corresponding tours. This incremental
evaluation strategy is a key ingredient in achieving good performance for large instances.
      The following snippet shows how the cost variation associated with a two-opt move can
be computed efficiently by considering only the edges affected by the move.


  1 size_t prev = (i == 0) ? n_nodes - 1 : i - 1;
  2 size_t next = (j == n_nodes - 1) ? 0 : j + 1;
  3
  4 return costs(tour[i], tour[next]) + costs(tour[prev], tour[j])       /* created edges */
  5         - costs(tour[j], tour[next]) - costs(tour[prev], tour[i]); /* broken edges     */

            Listing 10: Partial implementation for the two-opt delta computation.

Since the tour is represented as an open sequence but evaluated as a cycle, the indices
immediately preceding 𝑖 and following 𝑗 are handled with wrap-around logic. The returned
value corresponds to the delta of the move, computed as the cost of the newly added edges
minus the cost of the removed ones.

4.1.5 Exploration Strategies: Selecting the Next Solution
Exploring the solution space requires a clear policy for selecting which neighbor to move
to at each iteration. In our setting, this choice reduces to how two-opt moves are evaluated
and accepted. One option is a First Improvement strategy, which scans the neighborhood
and immediately applies the first move that yields a cost reduction, without inspecting the
remaining neighbors. An alternative is Best Improvement, where all feasible two-opt moves
are evaluated, the associated cost deltas are computed, and the move leading to the largest
improvement is selected. While hybrid strategies are possible, we restrict attention to these
two classical approaches for clarity.
      In practice, we predominantly rely on Best Improvement. Although it has a higher
theoretical complexity due to the full neighborhood evaluation, it consistently performed
better in our experiments. Moreover, the systematic traversal of neighbors tends to exhibit
better cache locality, partially offsetting its higher asymptotic cost, whereas First Improve­
ment may terminate early but does so in a less predictable memory-access pattern. From an
implementation perspective, the two strategies are interchangeable: all routines responsible
for neighbor selection are parameterized by a simple policy flag, allowing the exploration
strategy to be switched with minimal code changes.
      Starting from the initial solution constructed by the nearest-neighbor heuristic, the
algorithm iteratively explores the neighborhood of the current tour using two-opt moves.
At each iteration, a neighboring solution is selected according to the chosen improvement
policy and, if it yields a strictly lower cost, it replaces the current solution. This process is
repeated, progressively refining the tour as long as improving moves can be found.



                                               33
   The procedure naturally terminates when a full neighborhood exploration fails to
identify any improving two-opt move. At this point, the current tour is locally optimal with
respect to the chosen neighborhood structure, meaning that no single two-opt reversal can
further reduce the total travel cost. Reaching such a local optimum signals the end of the
local search phase, at which point the algorithm either stops or, in more advanced variants,
hands control to diversification mechanisms such as restarts or tabu strategies, which we
will explore in what follows.


4.2 A First Attempt to Metaheuristics: Tabu Search
To overcome the limitation of local optima, we employ a Tabu Search strategy. This
approach allows the algorithm to perform non-improving moves, enabling it to escape local
minima. Each move performed is recorded in a tabu list, which temporarily forbids reversing
that move for a specified number of iterations, known as the tabu tenure. By controlling
which moves are prohibited, the search explores new regions of the solution space while
avoiding simple cycles, increasing the chance of finding better solutions beyond local optima.

4.2.1 Setting Up the Tabu List
To efficiently implement the tabu mechanism, we developed a dedicated TabuList class that
internally represents the tabu list as a hash map. Rather than storing individual nodes, the
tabu list keeps track of edges—specifically, the pairs of nodes involved in a move—so that
a two-opt swap cannot be immediately undone. This is a stronger restriction than merely
tracking nodes, as it directly prevents reversing a particular change to the tour. Each edge
(𝑖, 𝑗) is treated symmetrically, meaning that (𝑖, 𝑗) and (𝑗, 𝑖) refer to the same entry in the
tabu list. The hash function used to compute the key to use in the map (which is more
convenient than using a std::pair) yields a unique identifier for each edge: for 𝑖 < 𝑗, the
hash is computed by placing 𝑖 in the upper 16 bits, and 𝑗 in the lower 16 bits of a 32-bit
integer (and vice versa if 𝑖 > 𝑗).


   1 inline u32 hash(u16 i, u16 j) const {
   2     return i < j ? ((u32)i << 16) | (u32)j : ((u32)j << 16) | (u32)i; /* bit ops */
   3 }

prevents cycles and adds diversity to the search. Checking whether a move is tabu simply
queries the hash map for the edge and compares the stored expiration iteration to the

To maintain efficiency and control memory usage, the tabu list periodically executes a
cleanup procedure that removes expired entries once the map grows beyond a threshold
proportional to the tabu tenure and oscillation range. This keeps the data structure
compact, ensuring fast lookups while avoiding excessive memory consumption.

4.2.2 Defining the Aspiration Criteria
In the context of tabu search, we occasionally allow moves that are otherwise tabu to be
performed if they satisfy a condition known as an aspiration criterion. In our implemen­
tation, a tabu move can be accepted if it results in a solution whose cost is better than
the best solution found so far. This mechanism ensures that the algorithm does not miss
promising improvements simply because a move is temporarily forbidden, allowing the
search to escape local optima and continue progressing toward higher-quality solutions.

4.2.3 Dynamic Tabu Search Behavior
Our tabu search implementation introduces a dynamic policy that adjusts the strictness of
tabu restrictions depending on the search state. Specifically, a two-opt move is normally
considered tabu based on the edges involved, but the exact rule depends on a flag
use_strong_policy. When the strong policy is active, a move is forbidden even if just one of

the two edges is currently tabu, enforcing a stricter exploration. Conversely, in the default,
more forgiving mode, a move is only considered tabu if both edges are tabu.
   The use_strong_policy flag itself is not static and it can be triggered dynamically when
the search detects stagnation. For instance, if no improving solution has been found for a
number of iterations, a “kick-start” mechanism sets use_strong_policy = true for a limited


                                                35
duration. This temporarily forces the search to accept worse solutions and explore less-
visited parts of the solution space, increasing diversity and (possibly) helping escape local
optima.

4.2.4 Improving the Performance: Multi Start & Parallelization
To strengthen the tabu search and increase the chances of finding high-quality solutions, we
implemented multi-start strategies that exploit multiple initial solutions. Multi-start allows
us to bypass the dependency on a single starting point, which is especially useful since the
quality of the initial nearest-neighbor solution can vary significantly. In our implementation,
we developed two metaheuristics that both incorporate multi-start but handle it differently,
trading off parallelism, filtering, and complexity.
   The first approach is parallel funneling multi-start. Here, we generate a large number
of candidate initial solutions using nearest-neighbor starting from different nodes, option­
ally refining them with a preliminary two-opt. We then select the best subset of these
candidates, called survivors, and run independent tabu searches for each in parallel. This
design maximizes exploration by leveraging modern multi-core hardware while funneling
computational effort toward the most promising candidates. The parallel execution ensures
that multiple searches proceed simultaneously without waiting on each other, improving
wall-clock time efficiency.
   The second approach is sequential pruning multi-start, which performs the multi-start
process one candidate at a time. After generating an initial solution with nearest-neighbor,
the algorithm evaluates its quality relative to the best solution found so far. If the initial
tour is worse than a configurable threshold compared to the current best, the candidate
is pruned and its tabu search is skipped. This filtering step compensates for the lack of
parallelism by avoiding expensive tabu searches on obviously poor initial solutions. While
simpler to implement, the sequential pruning approach is sensitive to the threshold choice:
too lenient and computational resources are wasted on low-quality candidates; too strict
and potentially strong solutions might be ignored early, especially due to “luck” in the order
of starting nodes.
   Both approaches exploit multi-start diversity, which increases the likelihood of escaping
local optima, and they naturally integrate with the tabu search framework. Parallel
funneling emphasizes exploration across promising candidates in a parallelized environment,
whereas sequential pruning is simpler and focuses computational effort selectively, with the
trade-off of being sensitive to threshold tuning and the sequential ordering of starts.
   The code for the two algorithms is not too complex, but it’s quite long, so it is omitted
from the report and remains available in the codebase for inspection.
   Up to this point, our focus has been on combining two-opt local search with tabu search
and multi-start strategies, applying them in a “flat” manner to the entire instance. These
approaches are largely general-purpose: they do not leverage any structural information
specific to the problem, beyond exploiting symmetry in the evaluation of moves. This
works well for small and medium instances and even produces reasonable results on larger

                                              36
instances, but the computational cost grows quickly as the instance size increases. To push
efficiency further, it becomes natural to consider the intrinsic structure of our instances
(components such as grids, lines, or clusters of holes) and exploit them hierarchically. The
next section introduces a hierarchical tabu search, which applies the metaheuristic in a way
that respects and leverages this clustered structure, improving scalability and execution
times for large, structured instances, at the cost of possibly slightly worse solutions. We
will analyze the trade-offs and see whether it is worth it or not.


4.3 Exploiting the Problem Structure
As observed in previous sections, particularly during the discussion on structured instance
generation and the visualization of sample boards, it is evident that many boards naturally
exhibit a clustered structure. These clusters, corresponding to components such as grids
or lines of holes, are too significant to ignore. The idea behind exploiting the problem
structure is to adopt a hierarchical approach, similar to divide-and-conquer, in which the
optimization is performed at two levels: intra-cluster, to find efficient tours within each
cluster, and inter-cluster, to determine the order in which clusters are visited. The tours
within and between clusters are then carefully stitched together to form a complete, feasible
global tour. This approach aims not only to preserve or improve solution quality but also
to achieve substantial speedups compared to the current flat metaheuristic that treats the
instance as unstructured.

4.3.1 Clustering Algorithm Selection
When deciding how to identify clusters in our PCB instances, we considered several common
clustering algorithms: K-means, DBSCAN, and hierarchical clustering. After testing these
approaches, we opted for K-means due to its simplicity, robustness across different board
configurations, and efficiency.
   In contrast, DBSCAN, which relies on density-based separation, proved less suitable
because many points in our structured instances would be considered noise, as the hole
distribution does not always form clearly dense regions. Hierarchical clustering was also
considered, but its higher computational cost and complexity made it less attractive for
larger instances, especially when repeated clustering would be required in the hierarchical
tabu search framework. Overall, K-means offered a practical balance of speed, consistency,
and reliable cluster identification for our problem.
   To automatically select the number of clusters K for each instance, we rely on the elbow
method based on the silhouette score, restricting the search to the range [2,32]. This allows
the algorithm to adaptively estimate a reasonable number of clusters without requiring
manual inspection for every board. The clustering itself is implemented in Python using
the scikit-learn library, which provides a convenient and efficient interface for K-means. It
is important to note that, although clustering is an integral part of the hierarchical tabu
search, we ignore its computation time in our performance analysis. This decision is justified


                                             37
for two reasons: first, it simplifies the integration with the C++ metaheuristic, avoiding cross-
language timing issues; second, K-means is extremely fast, and even for moderately sized
instances, its execution contributes negligibly to the overall runtime of the hierarchical
tabu search.
   A brief disclaimer is in order: we did not implement K-means ourselves, nor the silhou­
ette-based estimation for the number of clusters. Instead, we relied on external libraries and
existing functionality provided by Python’s scikit-learn, leveraging their tested and efficient
implementations to handle clustering and K selection automatically.
   A comparison between the aforementioned clustering algorithms is visually presented
in Figure 5. Beyond being very fast, K-means also provided the most balanced number of
clusters, while DBSCAN almost always produced too few, and Hierarchical Clustering too
many.
   For the sake of completeness, and to allow using our hierarchical tabu search also in
general cases, we tried applying these clustering algorithms to random instances that by
nature are not clustered. The results, however, were surprisingly good, both in terms of
how the clusters look visually, as seen in Figure 6, and also in terms of performance of the
actual implementation of our algorithm.
   The goal of our hierarchical approach is to exploit the natural clustered structure of the
board. By splitting the optimization into two phases (first focusing on intra-cluster paths
and then on inter-cluster connections) we aim to capture local geometric patterns more
efficiently. It is important to stress that this hierarchical approach is more restrictive than
the flat tabu search. By forcing the algorithm to solve each cluster independently and then
connecting them, we are limiting the solution space, which means the global optimum could
be excluded, especially if the clustering happens to be “unlucky” for a given instance. The
goal is therefore not necessarily to outperform the flat tabu search in solution quality, but
rather to achieve solutions that are reasonably close to optimal while drastically reducing
computation time, leveraging the structure of the problem to make the search more efficient.

4.4 A Second Attempt: Hierarchical Tabu Search
A major advantage of the hierarchical approach is that we can reuse much of the code
from the flat tabu search. Essentially, the problem is decomposed into smaller sub-instances
corresponding to the holes in each cluster. Each sub-instance can then be solved indepen­
dently using the same flat tabu search already developed, with the key benefit that smaller
instances are much faster to optimize. The same principle applies when determining the
order in which clusters themselves are visited, although this requires a careful definition of
inter-cluster distances to treat clusters geometrically. The truly novel part of the hierarchical
tabu search is the logic that allows us to stitch together the intra-cluster and inter-cluster
solutions into a single feasible tour. This step is both essential and non-trivial, and it is
what enables the hierarchical method to exploit the problem structure effectively.

4.4.1 Inter-Cluster Optimization
For the inter-cluster optimization, a crucial step is deciding how to represent clusters
geometrically. Our first attempt relied on centroids, treating each cluster as a single point
at the average location of its holes. While this is conceptually simple, it performed poorly in
practice. The centroid-based ordering often produces highly suboptimal cluster sequences,
because clusters are rarely compact: two clusters whose centroids are far apart may have
boundary points that are very close, allowing a cheap connection that the centroid-based
approach completely misses.
   To address this, we switched to a minimum-boundary distance approach. Here, the
distance between two clusters is defined as the shortest distance between any pair of points,
one from each cluster. This method better captures the true geometric relationship between
clusters and leads to more reasonable inter-cluster orderings. The downside is higher
computational complexity, since all pairwise distances between points of different clusters
must be evaluated. However, this cost is incurred only once at the beginning, effectively
as part of instance preprocessing, and does not affect the runtime of the hierarchical tabu
search itself. This trade-off was deemed acceptable, as it dramatically improves the quality
of inter-cluster sequencing without slowing down the actual optimization.
   For the actual hierarchical optimization, we leverage the sequential pruning multi-
start approach introduced earlier, but applied at the cluster level. Specifically, we generate
multiple inter-cluster tours, aiming to obtain the top‑𝑘 cluster orderings, where 𝑘 is a
tunable parameter.
   Each ordering is treated as a separate starting point for stitching the intra-cluster tours
computed with the flat tabu search. This allows us to explore different ways of connecting
clusters, compensating for the fact that even the minimum-boundary distances do not
guarantee a globally optimal inter-cluster sequence. By trying multiple high-quality cluster
orderings, the hierarchical approach combines local intra-cluster optimization with multiple
inter-cluster connection strategies, increasing the likelihood of finding a near-optimal global
tour while keeping the computational cost manageable.


                                               41
4.4.2 Intra-Cluster Optimization
For intra-cluster optimization, we treat each cluster as a standalone instance and apply the
sequential pruning multi-start tabu search within it. The key advantage here is that, since
clusters are independent at this stage, we can parallelize the optimization across clusters,
computing the best intra-cluster tours simultaneously. Each cluster’s tour is optimized
locally, benefiting from the small size of the cluster (faster tabu search), while maintaining
the flexibility to later combine these tours according to the multiple inter-cluster orderings
generated in the previous step. This parallelized intra-cluster optimization significantly
reduces runtime without compromising the quality of the cluster-level solutions.

4.4.3 Stitching the Cluster Tours to Obtain a Global Tour
Clusters are solved as standalone subproblems, meaning that the heuristic produces a closed
tour for each cluster. When assembling a global tour, however, each cluster tour must be
converted into an open path that can be connected to the preceding and succeeding clusters
in a feasible and cost-effective way. This transition from local closed tours to a global open
structure is non-trivial, both conceptually and algorithmically.
   A first natural idea is a greedy constructive approach. Given an ordering of clusters,
one could select, for each cluster, an entry point that minimizes the distance from the exit
point of the previous cluster. This can be implemented by simply rotating the cluster tour
so that it starts at the node closest to the previous cluster’s exit. While appealing in its
simplicity, this strategy performs poorly in practice.
   The underlying issue stems from the fact that each cluster tour is optimized in isolation.
A locally optimal tour is typically tight: the first and last nodes are close to each other.
This is precisely the desired outcome when solving the problem locally, but it becomes
problematic when the tour must be embedded into a larger global structure.
   If the tour is simply rotated to select an entry point that minimizes the connection
cost to the previous cluster, the exit point is implicitly fixed as well. Since the tour is
a rigid cycle, the last node of the rotated sequence will also be close to the entry point,
and therefore close to the previous cluster rather than to the next one. As a consequence,
the connection to the following cluster is often long and costly. Trying to optimize for the
following cluster as well as the previous, by introducing lookahead, does not fully resolve
this issue, as the entry point of the next cluster is itself unknown at that stage, leading to
a chicken-and-egg dependency.
   A more effective approach exploits a simple but crucial observation: a closed tour within
a cluster becomes an open path in the global tour. Rather than rotating a rigid cycle,
we can break exactly one edge of the cluster tour and use its two endpoints as the entry
and exit points of the cluster. This operation introduces two free endpoints, which can be
assigned asymmetric roles: one optimized for the connection to the previous cluster, and
the other optimized for the connection to the next. In symmetric instances a given tour
can be traveled in two directions (with no difference in cost), allowing each of the two end
nodes of a single edge to assume the role of either the entry or the exit node. This gives us

                                             42
an additional degree of freedom: since we can decide which end node (of some fixed edge)
to use as the entry node and which one to use as the exit node, we can just select as entry
node the one that is closest to the previous cluster, or select as exit node the one that is the
closest to the next cluster. Note that we cannot hope to select two nodes satisfying both
conditions, as we are constrained to choosing two nodes that are actually connected by an
edge in the tour within the current cluster.
   The central question then becomes which edge to break. Choosing an edge at random
is clearly suboptimal. Instead, a clever idea is to break the edge whose removal yields the
best overall connection to the surrounding clusters. Concretely, for a cluster positioned
between two others in the global ordering, we evaluate each edge of its tour as a candidate
break point.
   Although this requires examining all edges of the cluster tour, the computational cost
remains low in practice. Clusters are significantly smaller than the full instance, and the
cost variation induced by breaking an edge can be computed in constant time 𝒪︀(1), by
subtracting the cost of the removed edge and adding the costs of the two new inter-cluster
connections. This makes the approach both effective and computationally efficient, while
properly accounting for the interaction between neighboring clusters.
   A final clarification is required regarding what it concretely means to connect the current
cluster to its predecessor and successor when evaluating which internal edge to break. For
the previous cluster, the situation is simple: the connection is determined by the last node
of the current global tour, which coincides with the exit node of the cluster optimized in the
preceding iteration. This node is already fixed and can be used directly when evaluating
candidate break points in the current cluster.
   The connection to the following cluster, however, is less straightforward, as that cluster
has not yet been optimized. This introduces an unavoidable lookahead component. For
each candidate edge to be broken in the current cluster, and for each of the two possible
assignments of its endpoints as entry and exit, we approximate the future connection by
considering the node in the following cluster that is closest to the candidate exit node.
   Although approximate, this strategy yields a solution that is sufficiently informative for
decision-making. Moreover, it has a beneficial side effect: by selecting an exit node that is
already close to some node in the next cluster, we implicitly provide that cluster with a
favorable incoming connection. When the following cluster is subsequently optimized, this
advantage naturally propagates, often resulting in a short and well-aligned inter-cluster
bridge. In this way, the limited lookahead not only guides the current decision but also
facilitates the optimization of the next cluster, without incurring the full cost of a global
inter-cluster search.
   The underlying logic of the algorithm is notably elegant: a single, well-chosen edge
removal transforms an overconstrained local tour into a flexible component that can adapt
optimally to its global context. However, the code to implement this is quite long. For this
reason, rather than presenting lengthy and distracting source code, we provide only a high-



                                               43
level pseudocode description that captures the essential mechanics of the method while
keeping the focus on the algorithmic insight rather than its implementation overhead.


     1   Initialize 𝑇global with the tour of the first cluster 𝐶1
     2   𝑢curr ← last node of 𝑇global
     3   for each cluster 𝐶𝑘 in 𝐶ordering (from 𝑘 = 2 to 𝑁clusters ) do

4.4.4 Polishing the Stitched Global Tour
After the stitching phase, we apply a final polishing step to the global tour by running a
standard best improvement two-opt local search on the entire instance. The goal is to allow
for small adjustments and refinements that were not visible when optimizing each cluster
in isolation, but that become apparent once the full tour structure is fixed. Importantly,
this operation does not disrupt the overall organization of the solution: due to the way the
global tour is constructed, the solution lies deep inside a local optimum basin, and large-
scale rearrangements are extremely unlikely. As a result, the two-opt phase mainly smooths
minor inconsistencies at cluster boundaries and occasionally recovers small improvements.



                                                 44
4.4.5 Parameter Configuration & Tuning
After implementing the algorithm, and before being able to exploit it effectively, it is
necessary to select appropriate values for the parameters that characterize its behavior. In
our case, the most critical parameter of the tabu search is the tabu tenure, which determines
for how many iterations a move remains forbidden before becoming available again. The
optimal value of the tabu tenure strongly depends on the type and size of the instances
under consideration. Small instances often benefit from low or moderate tenure values,
while larger instances, which admit a vastly larger neighborhood of possible moves, typically
require higher tenures to avoid short cycling and premature looping.
   Since no single value can be fixed once and for all, the tabu tenure must be actively tuned
by testing different configurations on a common set of training instances and evaluating
their overall performance. This tuning phase is one of the most important steps in the
design of a tabu-search-based heuristic, as it often makes the difference between obtaining
a good solution and settling for a merely acceptable one. Ideally, a reasonable amount of
time should be devoted to this phase, for instance through a grid search over plausible
parameter ranges.
   Our implementation also includes a dynamic tabu tenure mechanism, where the tenure
oscillates around a default value within a prescribed range. This introduces an additional
parameter that would, in principle, require tuning. For simplicity, we fixed the oscillation
amplitude to half of the base tabu tenure. This choice is admittedly arbitrary and not
supported by a strong theoretical justification. While it is likely suboptimal for certain
classes of instances, it represents a pragmatic compromise aimed at limiting the overall
tuning effort.
   Beyond the tabu tenure, several additional parameters arise from the overall heuristic
framework rather than from the tabu search itself. A notable example is the number of
multistart initializations. Choosing too few starting solutions increases the risk of getting
trapped in poor regions of the solution space, while choosing too many can lead to redun­
dancy and unnecessary computational effort. As with the tabu tenure, the appropriate
number of starts depends on the instance size and structure. Small instances typically
require only a limited number of starts, whereas larger instances benefit from a broader
exploration of the solution space. In this context, it is also important that the starting
solutions be sufficiently diverse: selecting starting points that are too similar undermines
the purpose of multistart.
   Another parameter is the threshold used for sequential pruning, which determines when
an initial solution generated by the nearest-neighbor heuristic is discarded because it is
excessively worse than the best solution found so far. This parameter is particularly delicate,
as it does not depend solely on instance size: even instances of identical size may require
different thresholds depending on their geometric structure. For this reason, a conservative
choice is preferable, allowing slightly worse initial solutions to pass the filter in order to
avoid prematurely discarding promising regions of the search space.


                                             45
   Finally, the maximum number of tabu search iterations and the associated patience
parameter, defined as the number of consecutive non-improving iterations tolerated before
termination, play a crucial role. Increasing these values generally improves solution quality,
but only up to a point. Beyond that, improvements become marginal and are no longer
justified by the additional computational cost, leading to diminishing returns.
   As this discussion makes clear, parameter tuning is inherently challenging, especially
when multiple interacting parameters are involved and each admits a wide range of plausible
values. While a comprehensive grid search would be the most reliable approach, it is
also computationally expensive. For this reason, we deliberately limit the time spent on
parameter tuning, accepting a potential loss in solution quality as a necessary trade-off.
In practice, we select parameter values as simple functions of the instance size, without
attempting an exhaustive optimization of the parameter space.


4.5 Running the Heuristic Solver
Once the heuristic framework has been fully implemented and its parameters selected, we
can turn to the execution of the heuristic solver. At this stage, the C++ code operates as
a self-contained program that reads a preprocessed instance (the cost-matrix .dat file and
the cluster distribution .clusters file) and applies tabu search (with the different variants
we have explored) to compute a high-quality routing solution.
   As for the exact solver, compilation and execution follow a Makefile-based workflow.

4.5.1 Compilation
Before running the heuristic solver, the code must be compiled. The implementation requires
a C++ compiler supporting at least the C++11 standard, such as g++. Unlike the SCF solver, no
external optimization libraries are required, as all heuristic components are implemented
directly in C++.
   The compilation is handled analogously to that of the SCF Solver, using make. The
appropriate commands can be found in 3.4.1.
   If the compilation completes successfully, the bin/ directory will contain the executables
single_instance_heuristic_solver and batch_heuristic_solver.


4.5.2 Solving an Instance
To solve a single instance, the single_instance_heuristic_solver executable is used. The
general command-line syntax is as follows:

  $ ./bin/single_instance_heuristic_solver <filename.dat> <filename.clusters> \
        [--verbose <0|1|2>] [--average <n_runs>] [--print-tours]


The program arguments have the same meaning as in the SCF Solver. The only differences
are that our heuristic solver also requires the .clusters file in order to be able to run the
hierarchical tabu search and that an additional optional parameter --print-tours can be
provided, in order to also print the tours corresponding to the different algorithms.

                                             46
4.5.3 Batch Solver
The batch solver provides an automated way to solve multiple instances in a single
execution. Instead of operating on a single instance (pair of .dat and .clusters files), the
program takes as input a pair of directories containing the corresponding files for the cost-
matrix and the clusters of many instances. The command syntax for the executable is the
following:

  $ ./bin/batch_heuristic_solver <costs_dir> <clusters_dir> \
        [--average <n_runs>] [--output <filename>]


The two mandatory arguments specify the input directories. The first, costs_dir, must
contain the .dat files describing the cost matrices of the instances. The second, clusters_dir,
must contain the files describing the cluster decompositions associated with those same
instances. The solver matches instances across the two directories by filename, ignoring the
file extension. In other words, each cost matrix file is paired with the cluster file that shares
the same base name.
   For example, suppose the directory costs_dir contains the files instance_01.dat and
instance_02.dat, while the directory clusters_dir contains the files instance_01.clusters and

instance_02.clusters. In this case, the solver will correctly associate instance_01.dat with

instance_01.clusters, and instance_02.dat with instance_02.clusters, and then process each

matched pair as an independent instance.
   The optional --output parameter specifies the name of the file where the aggregated
results are written. If no output filename is provided, the solver automatically creates a file
named results.dat in the working directory and writes all results to it.

This chapter presents the experimental evaluation of the methods introduced in this
work. We begin by describing the benchmark instances and the experimental setup used
throughout the study, clarifying how data were generated and grouped. We then analyze the
performance of the exact SCF solver implemented in CPLEX, focusing on the computational
cost in terms of execution times. This is followed by an evaluation of the proposed heuristic
algorithms, highlighting their scalability and practical effectiveness on larger instances.
The two approaches are then compared directly, in order to assess the trade-offs between
optimality and efficiency. Finally, we move beyond aggregate numerical results and examine
the structure of the resulting tours, providing qualitative insights into how different methods
shape the final solutions.


5.1 Benchmark Instances and Experiments
Before presenting the experimental results, a brief disclaimer is in order. Over the course
of this work, the modeling and algorithmic choices accumulated to the point where the
space of possible instance configurations and parameter combinations became extremely
large. While this flexibility is powerful, it also makes it impractical to exhaustively explore
all possible benchmarks within reasonable time and computational limits. As a result,
compromises were unavoidable: there are infinitely many valid ways to construct benchmark
datasets, and the ones adopted here represent a careful but necessarily limited selection.
   We restrict our attention to two broad classes of instances: random instances and
structured instances. Random instances consist of holes placed uniformly at random on the
board. They serve as a useful worst-case baseline, allowing us to assess the raw performance
of both the exact and heuristic methods in the absence of exploitable structure. In this
sense, they reflect a generic routing problem rather than the specific industrial scenario
motivating this work. At the same time, they provide a reference against which the benefits
of structure-aware approaches can be evaluated.
   Structured instances, on the other hand, are designed to reflect the type of organization
encountered in realistic drilling boards, where holes are arranged according to repeated or
clustered patterns. While these instances are still simplified and optimized for experimental
clarity rather than full realism, they capture the operational structure that the proposed
hierarchical and clustering-based heuristics are intended to exploit. An additional advantage
is that such instances are easier to visualize and interpret, which proved valuable when
analyzing and comparing tour structures.
   For both instance types, multiple instances were generated and grouped by size,
measured in terms of the number of nodes. In the case of structured instances, a fixed
high-level configuration is used for all instances of the same size. Variability is introduced


                                              48
through randomized geometric placement of the components, ensuring that each instance
has a distinct spatial layout and topological structure. This allows us to obtain more robust
results while preserving structural consistency within each size class.
   More specifically, random instances were generated with sizes ranging from 10 to 100
nodes. All instances are embedded in a board of size 100 × 100 units with a drilling head
speed fixed at 100 units per second (along both axes, for simplicity).
   For structured instances, achieving the same level of granularity is inherently more
difficult. Even simple structured layouts require a minimum number of nodes, making
very small instances impractical or meaningless. For this reason, structured instances were
generated only at sizes 30, 50, 75, and 100. While less granular than the random case, these
checkpoints are sufficiently well distributed to capture the main performance trends.
   Two final remarks are important for interpreting the results. First, the choice of
instance sizes is constrained by computational resources and time limitations. The selected
benchmarks are suitable for comparative analysis but are not extensive enough to support
a detailed statistical study of fine-grained performance patterns. With more time and
resources, a significantly larger and more diverse dataset, including variations in board size
and density, would be desirable.
   Second, these limitations affect the exact and heuristic methods differently. Solving
instances larger than 100 nodes with the SCF model in CPLEX was deemed prohibitively
expensive, especially when averaging over multiple runs for robustness. However, restricting
the heuristic algorithms to the same size range would fail to demonstrate their intended
scalability. For this reason, additional large structured instances, with sizes up to 1000
nodes, were considered exclusively for the heuristic setting. For these instances, optimal
solutions are out of reach for CPLEX, and performance is instead evaluated relative to
the Held–Karp lower bound, computed using external tools. Consequently, the results for
large instances should not be interpreted in terms of absolute optimality gaps, but rather
as indicators of qualitative trends and scaling behavior as the instance size grows.

5.3 Heuristic Algorithms Performance
We now turn to the analysis of the heuristic algorithms developed throughout the previous
chapter. To keep the presentation readable and avoid repeating long algorithm names, we
introduce the following shorthand notation:
• F.S. → Flat Sequential Pruning
• F.P. → Flat Parallel Funneling
• H.S. → Hierarchical Sequential
• H.P. → Hierarchical Parallel
All four methods are heuristic solvers, differing in how they generate initial solutions, exploit
parallelism, and leverage the structure of the PCB board.
   The flat approaches operate directly on the full instance. In F.S., multiple initial
solutions are generated using a nearest-neighbor heuristic and refined sequentially via tabu
search; poor initial solutions are discarded early if their cost exceeds a fixed threshold
relative to the best solution found so far. F.P. extends this idea by first generating a large
pool of candidate initial solutions, selecting the top-𝑘, and optimizing them in parallel,
replacing early pruning with a funnel-style selection phase.




                                                   53
                             The hierarchical approaches explicitly exploit the board’s component structure. Both
H.S. and H.P. split the problem into layers: first, an ordering of clusters (components)
is optimized; then, tours are optimized within each cluster; finally, the cluster tours are
stitched together into a global tour using a smart stitching procedure.
                             As in the CPLEX SCF Solver evaluation, all heuristic variants are tested across a
range of benchmark instances, spanning both random and structured layouts, with sizes
ranging from 10 to 100 nodes. This allows us to assess not only raw performance, but also
how structural awareness and parallelization affect solution quality and scalability across
different problem regimes.
                             In this section, we compare the performance between the heuristic algorithms and use
the Held-Karp lower bound as reference for computing the gaps. The next section will
provide further insights with a comparison of the heuristic algorithms and the CPLEX
SCF Solver.

5.3.1 Speedup & (Pessimistic) Optimality Gaps
The evaluated heuristics are compared along two orthogonal dimensions: computational
efficiency and solution quality. Computational efficiency is measured as speedup relative to
the Flat Sequential pruning heuristic (F.S.), which serves as the baseline. Solution quality
is assessed through the average optimality gap with respect to the Held–Karp lower bound
for the Traveling Salesman Problem.
                             It is important to note that the Held–Karp bound constitutes a lower bound on the
optimal tour length. Consequently, the reported gaps represent upper bounds on the true
optimality gap. While this may lead to slightly pessimistic absolute values, it ensures a
consistent and fair comparison across all heuristics.

5.3.2 The Impact of Parallelization and Structural Decomposition
The experimental results clearly separate the effects of parallelization from those of struc­
tural problem decomposition. Parallelization alone, as implemented in the Flat Parallel
Funnel heuristic, provides moderate reductions in wall-clock time by exploiting multiple
cores, but it does not alter the underlying structure or complexity of the search space. As
a result, its performance gains remain limited.
   In contrast, hierarchical decomposition fundamentally reshapes the optimization prob­
lem by exploiting the spatial and structural organization of PCB layouts. By decoupling
inter-cluster ordering from intra-cluster routing, the effective problem size is significantly
reduced. When combined with parallel execution, these benefits compound, yielding the
strongest overall performance and indicating that hierarchy and parallelism address com­
plementary computational bottlenecks.
   This efficiency gain comes at the cost of solution quality. Hierarchical heuristics consis­
tently exhibit larger optimality gaps than flat approaches, a consequence of restricting the
search space once cluster boundaries and ordering are fixed, thereby excluding certain global
rearrangements of drill holes. However, this loss remains moderate and well controlled: even
for the largest instances, average gaps stay below 7.5%, while runtime reductions exceed
50×. Moreover, gaps are measured relative to the Held–Karp lower bound, suggesting that
actual solution quality may be better than reported.

5.4 Comparing SCF Solver and Our Metaheuristics
With a clear understanding of how the SCF formulation solved by CPLEX behaves as
instance size increases, and of the strengths and limitations of the heuristic approaches, we
can now perform a direct comparison between exact and heuristic methods.
   This comparison is intentionally restricted to small instances, for which CPLEX is able
to compute provably optimal solutions within reasonable time limits. Limiting the analysis
to this regime allows for a fair and meaningful evaluation, where heuristic solutions can be
assessed against true optima rather than against lower bounds or approximate references.
The goal is not to challenge the theoretical optimality of exact methods, but to quantify
how closely and how consistently heuristic approaches can approximate optimal solutions
when exact benchmarks are available.

5.4.1 Qualitative Comparison of Running Times & Optimality Gaps
The results of the comparison for random instances are shown in Figure 13, whereas those
of structured instances are shown in Figure 14.
   Across both random and structured instances, all heuristic methods outperform CPLEX
by several orders of magnitude in terms of running times. Flat heuristics already achieve
speedups on the order of 102 and 103 , while hierarchical methods reach speedups of 104
and 105 as instance size increases, which is the reason why we had to use the log scale in
the plots.
   In terms of solution quality, the comparison remains favorable for the heuristics. The
Held–Karp lower bound, used in the larger instance experiments, is already very tight for
the small sizes considered here: its gap from the true optimal value computed by CPLEX
never exceeds 1.5%. This confirms that the lower bound is a reliable reference point and
that the optimal solutions are not significantly worse than what the bound predicts. When
using the actual optimal objective as reference, the heuristic gaps shrink slightly compared
to the previous analysis, remaining consistently small across both random and structured
instances. This indicates that while the heuristics sacrifice some optimality in exchange for
speed, they still produce solutions that are very close to the true optimum in practice.

5.4.2 Using Exact Solvers as the Comparison Baseline
Although exact solvers such as CPLEX are not viable for large-scale PCB drilling instances,
their inclusion in the experimental evaluation serves two essential purposes. First, it
provides a ground-truth reference for solution quality on small to medium instances,
enabling an exact assessment of heuristic optimality gaps. Second, it allows the magnitude
of computational savings achieved by heuristic and hierarchical methods to be quantified
in absolute terms rather than solely relative to other heuristics.
   The results show that while exact optimization quickly becomes computationally infea­
sible, the proposed hierarchical heuristics achieve several orders-of-magnitude speedups
with only a modest and predictable loss in optimality, making the overall trade-off worth it.

5.5 Beyond Numbers: Tour Structure Comparison
So far, the evaluation has focused on numerical metrics such as objective value and relative
gap. These indicators are useful, but they do not fully capture what the solutions actually
look like in practice. In the real world, what matters is not only how small the cost is,
but also how the tour behaves on the board: whether it produces reasonable paths, avoids
unnecessary crossings, and respects the spatial structure of the instance.
   To address this, the final section goes beyond raw numbers and compares solutions
based on their tour structure. We visualize representative instances and directly compare
the routes generated by the SCF solver and our heuristic algorithms. This approach offers
a more operational perspective, showing how solutions would translate into real drilling
patterns rather than abstract cost values.
   This structural analysis is especially relevant because gap metrics can be misleading.
Relative gaps are highly sensitive to the scale of the objective: when the objective value is
small, even minor absolute differences can produce large percentage gaps. In our setting,
where the optimal values are often already small, the resulting gaps can look worse than
the actual quality difference in practice. By examining tour geometry, we can better assess
whether the heuristic solutions are genuinely inferior or simply numerically different in a
way that does not matter operationally.

5.5.1 Qualitative Analysis of Tours
To complement the quantitative results, Figure 15 visualizes the actual paths generated
by the different solvers on a representative structured instance with 100 nodes. This visual
inspection reveals the structural distinctiveness of each approach and provides intuition
into the sources of their respective optimality gaps.
   The exact solution produced by CPLEX exhibits a globally optimized structure, charac­
terized by smooth transitions, minimal crossings, and subtle interleavings between spatial
regions. Notice how it efficiently handles the transition between dense grids. It enters a
grid, traverses it systematically, and exits at the point closest to the next logical target.
This solution serves as a visual reference for optimal routing behavior.
   In contrast, the Nearest Neighbour heuristic produces a highly suboptimal tour, charac­
terized by sharp zig-zags and long, inefficient edges crossing the entire board.
   Its large optimality gap reflects the algorithm’s greedy nature and inability to recover
from early poor decisions, that look good locally but end up being poorly when considered
in terms of the global tour.
   Adding a further simple heuristic like two-opt yields enormous improvements, that allow
to reduce the gap from 24% (Nearest Neighbour) to 7%. Visually, the two-opt heuristic
successfully removes the long crossing edges seen in the raw nearest neighbour tour. The
path looks “cleaner” and locally efficient. However, compared to CPLEX, it still lacks
global coherence. While individual segments are optimized, the order in which the major
components (grids) are visited remains suboptimal, legacy of the poor nearest neighbour


                                             61
initialization that cannot be easily fixed. In this context, after some improvements, two-opt
gets stuck in local minima.
   Flat Tabu Search significantly improves tour quality by reducing crossings and eliminat­
ing many long edges. The possibility of exploring non improving solutions allows for better
exploration of the solution space, which eventually yields a good solution. Nevertheless, even
if the resulting tour closely resembles the optimal structure, some global rearrangements
remain unexplored, leading to a small residual gap.
   The Hierarchical Tabu Search explicitly exploits the clustered structure by separating
inter-cluster ordering from intra-cluster routing. This yields clean and well-organized tours
that closely follow the physical layout of the board. Although some global shortcuts present
in the exact solution are no longer accessible due to the imposed hierarchy (cluster ordering),
the resulting loss in optimality remains modest, and this comes with the huge benefit of
running times that are orders of magnitude faster than the Flat Tabu Search and the exact
CPLEX SCF Solver.
   Overall, the figure illustrates the central trade-off explored in this work: hierarchical
decomposition sacrifices a small amount of global optimality in exchange for dramatically
improved scalability and computational efficiency. The visual similarity between hierarchical
and optimal tours further supports the conclusion that this trade-off is well balanced and
practically acceptable for PCB drilling applications, especially when real-time computation
is important.


5.6 Possible Improvements
While the proposed hierarchical and flat heuristic pipeline achieves strong performance
in both computational efficiency and solution quality, several avenues exist for further
improvement. These opportunities span move design, parameter tuning, clustering strate­
gies, and integration with exact methods.

5.6.1 Enhanced Local Search Operators
The current implementation relies primarily on two-opt and, while computationally cheap
and effective for good improvements from a poor initial solution, sometimes it is insufficient
to restructure tours at a global scale.
   More expressive move operators could be introduced, including: three-opt, which would
allow the solver to correct more complex crossing patterns, or or-opt, which preserves partial
path structures while exploring new placements.

5.6.2 Reactive and Adaptive Parameter Tuning
The performance of pruning thresholds, tabu tenure lengths, and candidate list sizes
depends strongly on instance size and structure. Currently, these parameters are either fixed
or manually tuned in an approximate manner. Potential improvements include: adaptive
parameter schedules, where thresholds and tenures evolve during the search based on


                                             62
convergence behavior, and instance-aware tuning, using simple geometric features (e.g.,
number of clusters or cluster density) to select parameters dynamically.

5.6.3 Improving Clustering to Mitigate the Price of Hierarchy
The effectiveness of the hierarchical approach depends critically on the quality of the
clustering. Refining the clustering stage could reduce the rigidity of the hierarchical decom­
position and mitigate the loss of global optimality. As discussed in our analysis, the current
“smart lookahead” aims for the nearest neighbor in the next cluster, which can be unstable if
the target cluster rotates. Switching to Centroid-Based Targeting (aiming for the geometric
center of the next cluster) would provide a more stable directional heuristic, preventing the
solver from locking onto “outlier” nodes that result in poor entry angles.
   Another advanced possibility is Cluster Boundary Relaxation, in which we introduce a
post-processing step that allows nodes on the periphery of a cluster to swap into adjacent
clusters. This would maintain the speed benefits of hierarchy while fixing the most glaring
inefficiencies where the cluster border arbitrarily separates two nearby nodes.

5.6.4 Integrating Exact Methods
While exact solvers are impractical for large instances, they can still be leveraged strategi­
cally when the instance size is limited to small values, which is the case in most clusters for
the instances that we consider. We could integrate exacts solver in the pipeline to perform
local exact optimization inside a cluster, or to perform the optimization of the cluster
ordering, when the number of clusters is not prohibitively large.

5.6.5 Implementation-Level Improvements
Finally, additional gains may be achieved through implementation-level and parallel opti­
mizations. Asynchronous parallel search, improved memory locality, and hardware-aware
optimizations could further reduce runtime overheads, particularly for large instances.
These improvements would complement the algorithmic contributions by maximizing the
practical performance of the proposed framework.
   Overall, these directions highlight that the proposed pipeline is not a fixed heuristic but
a flexible and extensible framework. The results presented in this work establish a strong
baseline, while the identified improvements provide clear paths toward reducing optimality
gaps and enhancing robustness, reinforcing the suitability of hierarchical heuristic optimiza­
tion for large-scale PCB drilling problems.

5.7 Conclusions
This work set out to study the drilling path optimization problem from both an exact and
a heuristic perspective, with the explicit goal of understanding not only what solutions
can be obtained, but how they are obtained, how they scale, and what trade-offs they
impose in practice. Along the way, we deliberately complicated our own lives by introducing
multiple modeling choices, solver configurations, instance types, and algorithmic layers.
While occasionally painful, this complexity was essential to expose the strengths and
limitations of each approach in a controlled and meaningful way.
   We began by formulating the problem as a Traveling Salesman–like model and imple­
menting an exact solution using a Single-Commodity Flow (SCF) formulation solved by
CPLEX. This provided a reliable baseline and a correctness reference. The SCF model
proved robust, flexible, and fully agnostic to the semantics of the cost matrix, confirming
that the formulation and implementation were sound. However, as expected, exact opti­
mization rapidly became computationally expensive as instance size increased. Even though
near-optimal solutions are often found quickly, the effort required to prove optimality
dominates runtime, making exact methods impractical beyond relatively small instances.
   To overcome these scalability limits, we introduced a heuristic pipeline centered around
hierarchical decomposition and tabu search. Clustering allowed us to break large instances
into manageable subproblems, while tabu search provided an effective local optimization
mechanism within and across clusters. A significant portion of the work focused on the
nontrivial task of stitching cluster-level tours into a coherent global tour. The edge-breaking
strategy emerged as a key insight here: by transforming closed cluster tours into open paths,
we gained the degrees of freedom needed to optimize inter-cluster connections without
overconstraining the solution. While elegant in concept, the implementation reflects the
inherent complexity of balancing local optimality with global consistency.
   Crucially, looking beyond numerical gaps proved essential. Structural comparisons of
tours revealed that many heuristic solutions, despite having non-negligible relative gaps,
are operationally very close to the exact ones. In practical terms, the drill head follows
clean, sensible paths that differ little from the optimal tour. This reinforces the idea that
numerical optimality is not always the most meaningful metric, especially when objectives
are small and gaps become inflated by normalization.
   In the end, this work demonstrates that exact and heuristic methods are not competitors
but complementary tools. Exact solvers provide rigor, validation, and insight into problem
structure. Heuristics provide scalability, flexibility, and practical usability. By combining
both, we obtain not only good solutions, but also confidence in their quality and behavior.
   Before closing, an apology is in order for the sheer length of this work. It grew steadily,
much like the execution times we carefully plotted, fueled by too many instances, too many
metrics, and far too many hours spent staring at routes that refused to improve by even a
fraction of a unit. If nothing else, this report stands as proof that a large amount of time
was spent and, besides the quality of the results, we hope the effort will be appreciated.
